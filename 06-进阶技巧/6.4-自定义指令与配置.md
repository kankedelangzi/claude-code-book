# 6.4 自定义指令与配置

> 打造专属的 Claude Code，让它更懂你。

## 🎯 场景描述

每次都要重复说一些事：
- "用中文注释"
- "用 TypeScript"
- "代码要加错误处理"

可以让 Claude Code 记住这些偏好。

## 💡 核心思路

```
配置文件 + 自定义指令 = 个性化 Claude Code
```

## 📝 方法一：用户级配置

### 配置文件位置

```
~/.claude/config.json       # 全局配置
~/.claude/instructions.md   # 自定义指令
```

### 自定义指令示例

**~/.claude/instructions.md**

```markdown
# 自定义指令

## 代码风格
- 使用 TypeScript，避免 any
- 变量命名用 camelCase
- 注释用中文
- 函数要有 JSDoc 注释

## 偏好
- 错误处理用 try-catch
- 优先使用 async/await
- React 组件用函数式写法
- 样式用 Tailwind CSS

## 不要做的事
- 不要用 var
- 不要写过长的函数（超过 50 行要拆分）
- 不要硬编码配置
```

### 效果

以后每次对话，Claude Code 都会遵循这些规则，不用再重复说。

## 📝 方法二：项目级配置

### .claude 目录

在项目根目录创建：

```
project/
├── .claude/
│   ├── instructions.md   # 项目专属指令
│   ├── templates/        # 代码模板
│   │   ├── component.ts
│   │   └── api.ts
│   └── prompts/          # 常用 prompt
│       ├── review.md
│       └── test.md
```

### 项目指令示例

**.claude/instructions.md**

```markdown
# 项目指令

## 技术栈
- Next.js 14 + TypeScript
- Prisma + PostgreSQL
- Tailwind CSS

## API 规范
- 路由：/api/v1/xxx
- 响应格式：
  ```json
  {
    "success": true,
    "data": {},
    "error": null
  }
  ```

## 命名约定
- 组件：PascalCase（UserCard）
- 函数：camelCase（getUserById）
- 文件：kebab-case（user-card.tsx）
```

## 📝 方法三：代码模板

### 创建模板

**.claude/templates/component.ts**

```typescript
import { FC } from 'react';

interface {{ComponentName}}Props {
  // props 定义
}

/**
 * {{ComponentName}} 组件
 * 
 * @description 组件描述
 */
export const {{ComponentName}}: FC<{{ComponentName}}Props> = (props) => {
  return (
    <div className="">
      {/* 组件内容 */}
    </div>
  );
};
```

### 使用模板

```
用组件模板创建一个 UserCard 组件
```

Claude Code 会根据模板生成代码，保持风格一致。

## 📝 方法四：常用 Prompt 存储

### 创建 Prompt 文件

**.claude/prompts/review.md**

```markdown
请审查以下代码：

1. 检查是否有 Bug
2. 是否符合项目规范
3. 性能是否有优化空间
4. 安全问题

代码：
{{code}}

请用以下格式输出：
- 问题：xxx
- 建议：xxx
- 严重程度：高/中/低
```

**.claude/prompts/test.md**

```markdown
为以下函数生成测试用例：

函数：
{{function}}

要求：
- 覆盖正常情况
- 覆盖边界情况
- 覆盖错误情况
- 使用 Jest
```

### 使用存储的 Prompt

```
用 review prompt 审查 src/utils/format.ts
```

## 📝 实战案例：团队共享配置

### 创建团队配置仓库

```
team-claude-config/
├── README.md
├── instructions.md       # 团队通用指令
├── templates/
│   ├── react-component.tsx
│   ├── api-handler.ts
│   └── test-file.test.ts
└── prompts/
    ├── code-review.md
    └── generate-docs.md
```

### 团队成员使用

```bash
# 克隆到本地
git clone git@github.com:team/team-claude-config.git

# 复制到项目
cp -r team-claude-config/.claude your-project/
```

### instructions.md 示例（团队版）

```markdown
# 团队代码规范

## 语言
- 代码注释用中文
- 变量名、函数名用英文
- Git commit message 用中文

## React 组件
- 使用函数式组件
- Props 要定义 TypeScript 接口
- 样式用 Tailwind，避免 inline style

## API 接口
- RESTful 风格
- 统一响应格式
- 错误码参考错误码表（见 docs/error-codes.md）

## 测试
- 核心业务逻辑必须测试
- 使用 Jest + React Testing Library
- 测试覆盖率要求 > 80%

## Git
- 分支命名：feature/xxx, fix/xxx, refactor/xxx
- Commit 格式：`类型: 描述`（如 `feat: 添加用户登录`）
```

## ⚡ 技巧：环境变量管理

让 Claude Code 帮你管理环境变量：

**创建 .env.example**

```
# 数据库
DATABASE_URL=postgresql://user:pass@localhost:5432/db

# 认证
JWT_SECRET=your-secret-key
JWT_EXPIRES_IN=7d

# 第三方服务
SMTP_HOST=smtp.example.com
SMTP_USER=noreply@example.com
```

**告诉 Claude Code**

```
项目使用 .env 文件管理配置。
创建新功能时，如果需要新的环境变量：
1. 添加到 .env.example
2. 在代码中用 process.env.XXX 读取
3. 提醒我更新本地 .env 文件
```

## 🎓 学到了什么

1. **自定义指令让 Claude Code 更懂你的偏好**
2. **项目配置可以团队共享，保持风格一致**
3. **模板让代码风格统一**
4. **常用 Prompt 存储，不用每次重写**

## ⚠️ 注意

- 敏感信息不要写在配置文件里
- 团队配置要定期更新
- 配置文件要加入版本控制
- 指令不要写太多，关键的是几个规则
