# 4.2 日志分析案例

> 大海捞针？从海量日志中找到关键信息。

## 🎯 场景描述

你的服务挂了，同事丢给你一个日志文件：

```
[2024-01-15 09:23:11] INFO: Server started on port 3000
[2024-01-15 09:23:45] INFO: User login attempt: user123
[2024-01-15 09:23:45] WARN: Password mismatch for user123
[2024-01-15 09:23:46] INFO: User login failed: user123
[2024-01-15 09:24:01] ERROR: Connection timeout to database
[2024-01-15 09:24:03] ERROR: Request failed: /api/users
[2024-01-15 09:24:03] ERROR: Unhandled exception in handler
```

日志有几千行，你需要找出：
- 什么时间开始出问题
- 根本原因是什么
- 影响范围有多大

## ✨ Claude Code 做法

**直接丢给它：**

```
这是一个 Node.js 服务的日志文件，请帮我分析：
1. 找出所有 ERROR 级别的日志
2. 按时间顺序排列
3. 分析问题的根本原因
4. 告诉我可能的修复方向
```

Claude Code 会快速梳理出关键信息。

**按条件筛选：**

```
帮我统计：
- 有多少条 ERROR 日志
- 有多少条 WARN 日志
- 哪些接口最容易出错
```

**自定义分析：**

```
我想知道用户 "alice" 相关的所有操作日志，按时间排序
```

## 📂 常用日志分析场景

### 1. 找出异常高峰

```
统计每分钟的 ERROR 数量，找出异常高峰时间段
```

### 2. 分析接口性能

```
帮我提取所有接口的响应时间，找出最慢的5个接口
```

### 3. 用户行为追踪

```
按 userId 过滤，找出这个用户的所有操作记录
```

## 🔧 实用指令模板

```
这是一个日志文件，帮我：
1. 过滤出所有 ERROR 级别的日志
2. 按时间顺序排列
3. 给出问题总结
```

```
这个日志里有数据库连接错误，帮我分析：
- 第一次出现的时间
- 一共出现了多少次
- 之后有没有恢复
```

```
帮我统计这个日志文件中的请求量和响应时间分布
```

## 💡 日志分析技巧

**给 Claude Code 有效信息的格式：**

```
日志格式：[时间] [级别] 消息
技术栈：Node.js + Express + PostgreSQL
问题：服务响应很慢

这是日志片段：
[粘贴日志]
```

**如果日志太长：**

不要一次全丢，分段分析：

```
这是最近100行日志，帮我分析有什么问题
```

然后根据分析结果再深入。

## 🚀 进阶案例：复杂日志分析

### 案例 1：分析服务崩溃原因

**原始日志（5000+ 行）：**

```
[2024-01-15 09:20:01] INFO: Server started
[2024-01-15 09:20:02] INFO: Connected to database
[2024-01-15 09:21:15] INFO: Request: GET /api/users (200) 45ms
[2024-01-15 09:21:18] INFO: Request: POST /api/orders (201) 120ms
[2024-01-15 09:22:30] WARN: Database connection pool exhausted
[2024-01-15 09:22:31] WARN: Database connection pool exhausted
[2024-01-15 09:22:32] ERROR: Connection acquire timeout
[2024-01-15 09:22:33] ERROR: Query failed: SELECT * FROM orders
[2024-01-15 09:22:35] ERROR: Unhandled rejection: SequelizeConnectionError
[2024-01-15 09:22:36] INFO: Server shutting down
```

**给 Claude Code 的分析请求：**

```
服务在 09:22 崩溃了，帮我分析根因：

日志片段：
[粘贴上面的日志]

技术栈：Node.js + Express + Sequelize + PostgreSQL

请告诉我：
1. 问题是什么时候开始的
2. 根本原因是什么
3. 为什么会连锁反应
4. 怎么预防
```

**Claude Code 分析结果：**

```
问题时间线：
09:22:30 - 首次出现连接池耗尽警告
09:22:32 - 获取连接超时
09:22:35 - 未处理的 Promise 拒绝导致崩溃

根本原因：
数据库连接池耗尽（pool exhausted）

可能触发因素：
1. 某个查询没有释放连接（忘记 await 或没有正确关闭）
2. 并发请求量突然增大
3. 连接池配置太小

修复建议：
1. 检查所有数据库操作是否正确释放连接
2. 增加连接池大小：pool: { max: 20, min: 5 }
3. 加连接超时监控
4. 给未处理的 rejection 加全局捕获：
   process.on('unhandledRejection', (err) => { ... })
```

### 案例 2：性能瓶颈定位

**日志格式：**

```
[2024-01-15 10:00:01] INFO: [api] GET /api/products 200 1523ms
[2024-01-15 10:00:02] INFO: [api] GET /api/users 200 45ms
[2024-01-15 10:00:03] INFO: [api] GET /api/products 200 1680ms
[2024-01-15 10:00:05] INFO: [api] POST /api/orders 201 234ms
[2024-01-15 10:00:06] INFO: [api] GET /api/products 200 2100ms
```

**分析请求：**

```
帮我分析这个日志，找出性能问题：

1. 统计每个接口的平均响应时间
2. 找出最慢的接口
3. 分析是否有异常慢的请求

日志：
[粘贴日志]
```

**Claude Code 输出：**

```
接口性能统计：
/api/products  平均 1768ms  调用 3 次  ⚠️ 最慢
/api/users     平均 45ms    调用 1 次
/api/orders    平均 234ms   调用 1 次

问题分析：
/api/products 响应时间异常（1.5-2.1秒），远超正常范围

可能原因：
1. 数据量大，缺少索引
2. N+1 查询问题
3. 没有缓存
4. 数据库连接慢

建议：
1. 检查 products 表的索引
2. 加 Redis 缓存
3. 用 EXPLAIN ANALYZE 分析查询
```

### 案例 3：用户问题排查

**场景：用户反馈"下单失败"**

```
帮我找出用户 user_12345 在 10:00-10:30 的所有操作日志：

日志文件：
[粘贴完整日志]

请提取：
1. 该用户的所有请求
2. 是否有失败的操作
3. 失败的具体原因
```

**Claude Code 输出：**

```
用户 user_12345 操作记录：
10:05:12 - GET /api/cart - 200 - 获取购物车成功
10:05:18 - POST /api/orders - 500 - 下单失败
10:05:18 - ERROR: Insufficient inventory for product_789

失败原因：
商品 product_789 库存不足

建议：
1. 检查该商品当前库存
2. 前端应显示库存不足提示而非 500 错误
```

## 🔧 日志格式规范化建议

### 结构化日志格式

```javascript
// 推荐：JSON 格式日志
const logger = winston.createLogger({
  format: winston.format.json(),
  transports: [new winston.transports.File({ filename: 'app.log' })]
})

logger.info('Request processed', {
  method: 'GET',
  path: '/api/users',
  status: 200,
  duration: 45,
  userId: 'user_123'
})
```

输出：

```json
{"level":"info","message":"Request processed","method":"GET","path":"/api/users","status":200,"duration":45,"userId":"user_123","timestamp":"2024-01-15T10:00:00Z"}
```

### 给 Claude Code 的分析优势

结构化日志让 Claude Code 更容易分析：

```
这是一个 JSON 格式的日志文件，请：
1. 统计每个 path 的请求量和平均响应时间
2. 找出所有 status >= 400 的请求
3. 按 userId 分组统计请求量
```

## 📋 日志分析指令速查

| 场景 | 指令模板 |
|-----|---------|
| 找出所有错误 | `过滤出所有 ERROR 级别的日志，按时间排序` |
| 统计错误分布 | `统计每种错误类型出现的次数` |
| 性能分析 | `提取所有请求的响应时间，找出最慢的 10 个` |
| 用户追踪 | `找出 userId=xxx 的所有操作记录` |
| 时间范围 | `提取 09:00-10:00 之间的所有日志` |
| 关联分析 | `找出 ERROR 前后 10 行的日志，分析上下文` |
| 趋势分析 | `按小时统计请求量，画出趋势` |
