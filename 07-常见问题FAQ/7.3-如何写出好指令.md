# 7.3 如何写出好指令？

> 垃圾进，垃圾出。指令写得烂，代码也不会好。

## 🎯 核心原则

好指令 = **做什么** + **什么风格** + **边界在哪**

缺一个，结果就可能跑偏。

## 💡 三要素公式

```
✅ 好指令 = 明确目标 + 上下文 + 约束条件

❌ 差指令 = "帮我写个页面"
```

## 📝 拆解：一份好指令长什么样

### 要素 1：明确目标

```
# 差 ❌
帮我写一个用户管理页面

# 好 ✅
创建一个用户管理页面，包含：
- 用户列表（表格展示，支持分页，每页 20 条）
- 搜索框（按用户名搜索）
- 新增用户按钮
- 每行有编辑和删除操作
```

具体、可拆解、能验证——这就是好目标。

### 要素 2：提供上下文

```
# 差 ❌
帮我改一下登录接口

# 好 ✅
修改 src/api/auth.ts 中的 login 函数：
- 当前使用的是 session 认证
- 需要改成 JWT token 认证
- token 有效期 24 小时
- 使用 jsonwebtoken 库（已安装）
- 需要兼容现有的前端调用方式
```

告诉它**现状是什么**，**要去哪里**，**用什么工具**。

### 要素 3：设定约束

```
# 差 ❌
帮我优化这个函数

# 好 ✅
优化 src/utils/formatDate.ts 中的 formatDate 函数：
- 保持现有功能不变
- 不要引入新的依赖
- 优先考虑可读性，其次性能
- 需要通过现有的测试用例
```

约束是护栏，防止 Claude Code "过度优化"。

## 🔧 实战：指令对比

### 案例 1：创建 API

```
# 差 ❌
帮我写一个用户 API

# 好 ✅
在 Express 项目中创建用户相关 API：
文件路径：src/routes/user.ts
包含以下接口：
  GET    /api/users       - 获取用户列表，支持 ?page=1&limit=20
  GET    /api/users/:id   - 获取单个用户详情
  POST   /api/users       - 创建用户（需要 name, email, password）
  PUT    /api/users/:id   - 更新用户信息
  DELETE /api/users/:id   - 删除用户

技术要求：
- 使用 TypeScript
- 使用 Prisma ORM 操作数据库
- 统一响应格式 { success, data, error }
- 错误处理用 try-catch
- 参数验证用 zod（已安装）
```

### 案例 2：修复 Bug

```
# 差 ❌
这个函数有 bug，帮我修一下

# 好 ✅
src/services/orderService.ts 的 calculateTotal 函数有 bug：
- 当购物车为空时返回 NaN 而不是 0
- 当商品价格为负数时没有校验

修复要求：
- 空购物车返回 0
- 负价格抛出错误 "Invalid price"
- 不要改变现有的正常计算逻辑
- 加上对应的单元测试
```

### 案例 3：重构

```
# 差 ❌
这段代码太乱了，帮我重构

# 好 ✅
重构 src/components/UserProfile.tsx：
问题：
1. 组件超过 300 行，需要拆分
2. 有重复的格式化逻辑
3. 缺少错误状态处理

要求：
- 拆分成 UserProfile（主组件）+ UserAvatar + UserInfo 三个组件
- 提取公共的格式化函数到 src/utils/format.ts
- 添加 loading 和 error 状态
- 保持现有样式不变
```

## 🧩 不同任务的最佳指令模板

### 创建新功能

```
创建 [功能名称]，包含以下功能点：
1. [功能点 1]
2. [功能点 2]
3. [功能点 3]

技术要求：
- 文件路径：[路径]
- 使用 [技术/库]
- 遵循 [项目规范]
- 需要/不需要测试
```

### 修改现有代码

```
修改 [文件路径] 中的 [函数/组件名称]：

当前行为：[描述现状]
期望行为：[描述目标]

约束：
- 不要修改 [不该动的部分]
- 保持向后兼容 / 可以破坏性变更
- 使用 [指定的方式]
```

### 调试

```
[文件路径] 出现 [错误信息]：

复现步骤：
1. [步骤 1]
2. [步骤 2]

期望结果：[正确结果]
实际结果：[错误结果]

已尝试：[你做过什么]
相关代码：[贴出关键代码片段]
```

## ⚡ 高级技巧

### 技巧 1：分步指令

大任务不要一次性扔给它，拆成小步骤：

```
第一步：先帮我设计数据库表结构，我要你确认后再写代码
```

等它回复确认后：

```
好的，按这个结构生成 Prisma schema
```

再下一步：

```
基于这个 schema，创建 CRUD API
```

### 技巧 2：提供示例

```
帮我创建一个和 src/components/UserCard 类似风格的 ProductCard 组件
```

有参照物，风格就不会跑偏。

### 技巧 3：先让它思考

```
我要重构这个模块，先帮我分析一下有哪些方式，
列出每种方式的优缺点，不要直接写代码
```

让它先给方案，你选一个再让它实现。

### 技巧 4：设定角色

```
你是一个有 5 年经验的 React 开发者，
擅长性能优化和无障碍设计。
帮我 review 一下这个组件。
```

角色设定能让输出更聚焦。

## ❌ 常见错误

| 错误 | 问题 | 修正 |
|------|------|------|
| "帮我改一下" | 改什么？怎么改？ | 说清文件、位置、期望结果 |
| "像淘宝那样" | 太模糊 | 截图或详细描述具体功能 |
| "代码优化一下" | 优化什么？性能？可读性？ | 明确优化维度和指标 |
| 一次给 10 个任务 | 信息过载 | 每次一个任务 |
| 不给上下文直接贴代码 | 它不知道前后关系 | 说清文件位置、技术栈、已有代码 |

## 🎓 学到了什么

1. **好指令 = 目标 + 上下文 + 约束**，缺一不可
2. **越具体越好**，模糊是效率的敌人
3. **大任务拆小步**，每一步都确认再往下走
4. **提供参照物**，风格和结构就不会跑偏

## ⚠️ 注意

- 不要指望一次指令就完美，好的结果往往需要 2-3 轮对话
- 指令写不好不是你笨，是经验不够——多写几次就有感觉了
- 如果你发现自己反复说同样的要求，把它写进 `instructions.md`
