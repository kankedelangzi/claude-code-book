# 3.1 代码重构实战

> 重构不用慌，Claude Code 帮你逐块改造。

## 🎯 场景描述

你接手了一段"能用但不想看"的代码，想要重构但：

- 逻辑太乱，不敢动
- 测试覆盖率低，怕改坏
- 不知道从哪开始

**Claude Code 可以逐个函数、逐个文件地帮你重构，每一步都能验证。**

## ✨ 经典重构场景

### 场景 1：长函数拆分

**重构前：**

```javascript
function processOrder(order) {
  // 验证订单
  if (!order.items || order.items.length === 0) throw new Error('Empty order');
  if (!order.customerId) throw new Error('No customer');
  // 计算价格
  let total = 0;
  for (const item of order.items) {
    total += item.price * item.quantity;
  }
  if (order.discount) total -= order.discount;
  // 生成 ID
  const orderId = 'ORD-' + Date.now();
  // 保存订单
  db.orders.insert({ orderId, ...order, total, status: 'pending' });
  // 发邮件
  sendEmail(order.customerId, 'Order confirmed: ' + orderId);
  // 返回
  return { orderId, total };
}
```

**重构指令：**

```
把这个长函数拆成几个小函数：
- validateOrder(order) - 验证逻辑
- calculateTotal(items, discount) - 计算价格
- saveOrder(orderData) - 保存数据库
- notifyCustomer(customerId, orderId) - 发送通知
主函数 processOrder 只负责调用这些小函数。
```

**重构后：**

```javascript
function validateOrder(order) {
  if (!order.items?.length) throw new Error('Empty order');
  if (!order.customerId) throw new Error('No customer');
}

function calculateTotal(items, discount) {
  const subtotal = items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  return discount ? subtotal - discount : subtotal;
}

function saveOrder(orderData) {
  return db.orders.insert(orderData);
}

function notifyCustomer(customerId, orderId) {
  sendEmail(customerId, `Order confirmed: ${orderId}`);
}

function processOrder(order) {
  validateOrder(order);
  const total = calculateTotal(order.items, order.discount);
  const orderId = `ORD-${Date.now()}`;
  const orderData = { orderId, ...order, total, status: 'pending' };
  saveOrder(orderData);
  notifyCustomer(order.customerId, orderId);
  return { orderId, total };
}
```

**每一步都可单独测试**——改坏哪里，一目了然。

### 场景 2：嵌套回调 → async/await

**重构前（回调地狱）：**

```javascript
function getUserData(userId, callback) {
  db.getUser(userId, (err, user) => {
    if (err) return callback(err);
    db.getOrders(userId, (err, orders) => {
      if (err) return callback(err);
      db.getPreferences(userId, (err, prefs) => {
        if (err) return callback(err);
        callback(null, { user, orders, preferences: prefs });
      });
    });
  });
}
```

**重构指令：**

```
把回调改成 async/await 风格，保持相同的输入输出。
不要改变业务逻辑，只是改语法形式。
```

**重构后：**

```javascript
function getUserData(userId) {
  const user = await db.getUser(userId);
  const orders = await db.getOrders(userId);
  const preferences = await db.getPreferences(userId);
  return { user, orders, preferences };
}
```

### 场景 3：硬编码配置化

**重构前：**

```javascript
function sendNotification(userId, type) {
  if (type === 'email') {
    const user = db.users.find(userId);
    const email = user.email;
    // 发送邮件...
    smtp.send({ to: email, subject: '通知', body: '...' });
  } else if (type === 'sms') {
    const user = db.users.find(userId);
    const phone = user.phone;
    // 发送短信...
    sms.send({ to: phone, message: '...' });
  }
}
```

**重构指令：**

```
把发送逻辑抽象成策略模式：
- 提取每种通知类型的发送器
- 用配置对象管理类型到发送器的映射
- 支持运行时注册新的通知类型
```

**重构后：**

```javascript
const notifiers = {
  email: async (user, message) => smtp.send({ to: user.email, ...message }),
  sms: async (user, message) => sms.send({ to: user.phone, ...message }),
};

function registerNotifier(type, fn) {
  notifiers[type] = fn;
}

async function sendNotification(userId, type, message) {
  const user = await db.users.find(userId);
  if (!notifiers[type]) throw new Error(`Unknown type: ${type}`);
  await notifiers[type](user, message);
}
```

### 场景 4：重复代码提取公共函数

**重构前：**

```javascript
// 文件1: auth.js
function login(username, password) {
  const user = db.users.findOne({ username });
  if (!user) throw new Error('User not found');
  if (!bcrypt.compare(password, user.passwordHash)) throw new Error('Wrong password');
  const token = jwt.sign({ id: user.id }, process.env.JWT_SECRET);
  return { token, user: { id: user.id, username: user.username } };
}

// 文件2: admin.js
function loginAdmin(username, password) {
  const admin = db.admins.findOne({ username });
  if (!admin) throw new Error('Admin not found');
  if (!bcrypt.compare(password, admin.passwordHash)) throw new Error('Wrong password');
  const token = jwt.sign({ id: admin.id, role: 'admin' }, process.env.JWT_SECRET);
  return { token, user: { id: admin.id, username: admin.username } };
}
```

**重构指令：**

```
两个 login 函数逻辑几乎一样，差异只有：
1. 查找用户的表不同（users / admins）
2. 返回的 role 字段不同

提取一个通用函数 authenticate(tableName, entityType, credentials)，
专门处理密码验证和 token 生成。
```

## 🔄 重构工作流

```
1. 选定目标
   ├── 打开要重构的文件
   └── 告诉 Claude Code："这个文件有个 XX 问题，我想重构它"

2. 描述期望
   ├── "我想把它改成 XX 风格"
   └── "保持功能不变，只改结构"

3. 逐块验证
   ├── 每次改动后跑一下测试
   └── 确认功能不受影响

4. 提交代码
   └── git commit -m "refactor: [描述改动]"
```

## ⚠️ 重构安全守则

| 原则 | 说明 |
|------|------|
| **有测试再动** | 重构前先补单元测试，没有就写几个简单的 |
| **一次只改一件事** | 不要同时重构+加功能+修 bug |
| **保持功能不变** | 重构是改结构，不是改行为 |
| **频繁提交** | 每完成一个小重构就 commit，方便回滚 |

## 📝 小结

Claude Code 重构的优势：

- **逐块改造** — 不用一次看完全部代码
- **保持功能** — 改动前后行为一致
- **有据可查** — 每次 commit 记录清晰
- **降低心理门槛** — 不再害怕"动旧代码"

**重构的本质是沟通——你告诉 AI 要什么，它来执行。**
