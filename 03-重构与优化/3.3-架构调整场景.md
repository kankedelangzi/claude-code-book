架构调整往往是"大手术"：

- 从单体拆分为微服务

## ✨ 经典架构调整场景

### 场景 1：单体拆分为微服务

**背景：** 一个电商系统，所有功能在一个 Express 应用里，想要拆成独立服务。

**拆分策略：**

```
订单模块 → order-service (端口 3001)
用户模块 → user-service (端口 3002)
商品模块 → product-service (端口 3003)
```

**拆分指令（以订单服务为例）：**

```
从这个单体应用中提取订单相关代码，拆成独立服务 order-service：
1. 提取订单相关的路由、模型、中间件
2. 保留原有的数据库访问层（订单表操作）
3. 创建独立的 package.json 和入口文件
4. 服务只暴露必要的 API，不暴露内部实现
5. 添加健康检查接口 /health
```

**拆分后目录结构：**

```
microservices/
├── order-service/
│   ├── src/
│   │   ├── routes/orders.js
│   │   ├── models/order.js
│   │   └── index.js
│   ├── package.json
│   └── .env
├── user-service/
│   └── ...
└── api-gateway/
    ├── src/
    │   └── index.js    # 统一入口，路由转发
    └── package.json
```

**API Gateway 转发示例：**

```javascript
const express = require('express');
const { createProxyMiddleware } = require('http-proxy-middleware');

const app = express();

app.use('/api/orders', createProxyMiddleware({
  target: 'http://localhost:3001',
  changeOrigin: true
}));

app.use('/api/users', createProxyMiddleware({
  target: 'http://localhost:3002',
  changeOrigin: true
}));

app.use('/api/products', createProxyMiddleware({
  target: 'http://localhost:3003',
  changeOrigin: true
}));

app.listen(3000);
```

### 场景 2：REST → GraphQL

**背景：** REST API 端点过多，客户端需要多次请求才能拿到完整数据。

**迁移策略：** 渐进式迁移，不破坏现有 REST 接口。

**迁移指令：**

```
在现有 Express 项目中添加 GraphQL 支持：
1. 安装 apollo-server-express
2. 定义 typeDefs（用户、订单、商品的类型）
3. 定义 resolvers（复用现有的 service 层）
4. GraphQL 端点 /graphql，REST 端点保留
5. 两个端点共用同一套 service 代码
```

**迁移后代码：**

```javascript
const { ApolloServer, gql } = require('apollo-server-express');

const typeDefs = gql`
  type User {
    id: ID!
    name: String!
    email: String!
    orders: [Order]
  }

  type Order {
    id: ID!
    total: Float!
    items: [OrderItem]
    user: User
  }

  type Query {
    user(id: ID!): User
    users: [User]
    orders(userId: ID): [Order]
  }
`;

const resolvers = {
  Query: {
    user: (_, { id }) => getUserById(id),
    users: () => getAllUsers(),
    orders: (_, { userId }) => getOrders(userId)
  },
  User: {
    orders: (user) => getOrdersByUserId(user.id)
  },
  Order: {
    user: (order) => getUserById(order.userId)
  }
};

const server = new ApolloServer({ typeDefs, resolvers });
server.applyMiddleware({ app, path: '/graphql' });
```

**新旧接口共存，客户端逐步迁移到 GraphQL。**

### 场景 3：jQuery → React

**背景：** 老项目用 jQuery，想迁移到 React 但不想重写。

**迁移策略：** 逐个组件迁移，先从简单的开始。

**迁移指令：**

```
把这段 jQuery 代码改成 React 函数组件：
- 保持相同的 DOM 结构
- 保持相同的交互逻辑（点击切换、输入响应）
- 不改变样式，只改代码结构
- 用 useState 管理状态，不用 Redux
```

**jQuery 代码：**

```javascript
$('#tabs .tab').click(function() {
  $('#tabs .tab').removeClass('active');
  $(this).addClass('active');
  $('#content .panel').hide();
  $('#content .panel').eq($(this).index()).show();
});
```

**React 版本：**

```jsx
function Tabs({ tabs }) {
  const [activeIndex, setActiveIndex] = useState(0);

  return (
    <div id="tabs">
      {tabs.map((tab, i) => (
        <div
          key={i}
          className={`tab ${i === activeIndex ? 'active' : ''}`}
          onClick={() => setActiveIndex(i)}
        >
          {tab.label}
        </div>
      ))}
      <div id="content">
        {tabs.map((tab, i) => (
          <div key={i} className="panel" style={{ display: i === activeIndex ? 'block' : 'none' }}>
            {tab.content}
          </div>
        ))}
      </div>
    </div>
  );
}
```

**样式完全兼容，交互逻辑保持一致。**

### 场景 4：Java 版本升级

**背景：** 项目还在用 Java 8，想要升级到 Java 17。

**升级策略：** 渐进式，先让代码在 Java 17 上跑起来，再逐步使用新特性。

**升级指令：**

```
把这段 Java 8 代码升级到 Java 17：
1. 用 var 类型推断（Java 10+）
2. 用新的日期 API（java.time 已经在用）
3. 用 sealed class 替代枚举如果合适
4. 去掉过时的注解 @SuppressWarnings("unchecked")
5. 保持字节码兼容，不要用 preview feature
```

**升级后示例：**

```java
// Before (Java 8)
List<String> items = new ArrayList<>();
Map<String, Object> config = new HashMap<>();

// After (Java 17)
var items = new ArrayList<String>();
var config = new HashMap<String, Object>();

// Record (Java 16+)
public record UserDTO(String id, String name, String email) {}

// Pattern matching for instanceof (Java 16+)
if (obj instanceof String s && s.length() > 5) {
    System.out.println(s.toUpperCase());
}
```

## 🔄 架构调整工作流

```
1. 评估现状
   ├── 画出当前的架构图
   ├── 识别关键依赖和风险点
   └── 确定拆分/迁移边界

2. 设计目标架构
   ├── 画出目标架构图
   ├── 定义模块间的接口契约
   └── 制定迁移顺序（先易后难）

3. 逐模块迁移
   ├── 按顺序改造每个模块
   ├── 保持新旧接口共存
   └── 每步都验证功能正常

4. 清理旧代码
   ├── 确认迁移完成后
   └── 删除旧的兼容代码
```

## ⚠️ 架构调整注意事项

| 原则 | 说明 |
|------|------|
| **接口契约先定** | 模块间用什么接口通信，先定清楚 |
| **不破坏现有功能** | 新架构要能跑通所有旧场景 |
| **可回滚** | 每步迁移都能回退到上一步 |
| **小步快跑** | 不要憋大招，每次迁移一小块 |

## 📝 小结

架构调整的关键：

- **渐进式** — 不追求一步到位
- **接口稳定** — 模块间通过契约通信，不暴露内部
- **可验证** — 每步迁移后都要测试
- **有记录** — 记录迁移决策和踩的坑

**Claude Code 擅长执行具体的代码改动，但架构决策需要你来定。**
