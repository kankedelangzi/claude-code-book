# 3.4 遗留代码改造

> 没有文档、没有测试、没人敢动的老代码，Claude Code 帮你摸清套路、逐步改造。

## 🎯 场景描述

遗留代码的特点：

- 没有文档，不知道"为什么这么做"
- 没有测试，改了怕出问题
- 逻辑绕来绕去，看不懂
- 变量名是 a、b、c

**Claude Code 可以帮你：理解代码 → 补充文档 → 编写测试 → 逐步重构。**

## ✨ 改造工作流

### Step 1：让 Claude Code 读懂代码

```
帮我分析这个文件：
1. 总结每个函数的作用
2. 画出数据流向（输入什么、输出什么）
3. 找出关键的决策点（if/else 的逻辑）
4. 标出看不懂的地方，我来解释
```

**输出示例：**

```markdown
## 文件分析：order-service.js

### 函数清单
- `processOrder()` - 主处理函数，接收订单数据，调用下游服务
- `calcPrice()` - 计算订单价格，考虑折扣和会员等级
- `sendNotification()` - 发送订单通知（邮件/短信）
- `validateOrder()` - 验证订单合法性

### 数据流向
用户提交 → validateOrder → calcPrice → 保存数据库 → sendNotification → 返回

### 关键决策点
- 折扣：会员 9 折，订单 > 1000 再减 50
- 通知：vip 用户发邮件，普通用户发短信
- 状态：支付成功 → 配货中 → 已发货 → 已完成

### 不确定的地方
- `syncStock()` 这个函数看起来没用到？
- `retryCount` 是用来干什么的？
```

### Step 2：补充文档和注释

```
在每个函数上方添加 JSDoc 注释：
- @param: 参数说明
- @returns: 返回值说明
- @throws: 可能抛出的异常
- 简单描述函数的作用
```

**输出示例：**

```javascript
/**
 * 处理订单主流程
 * @param {Object} orderData - 订单数据，包含 items、customerId、paymentInfo
 * @returns {Promise<{orderId: string, total: number}>} 处理结果
 * @throws {ValidationError} 订单数据不合法
 * @throws {PaymentError} 支付失败
 */
async function processOrder(orderData) {
  // ...
}
```

### Step 3：编写测试

**先问清楚边界条件：**

```
这个函数有哪些边界情况需要测试？
1. 正常情况
2. 边界值（空数组、零值、最大值）
3. 异常情况（参数缺失、类型错误）
4. 并发情况（如果有共享状态）
```

**然后生成测试：**

```
为 calcPrice 函数生成 Jest 测试：
- 用 describe/it 结构
- 覆盖所有你说的边界情况
- 用 test.each 做参数化测试
```

**输出示例：**

```javascript
describe('calcPrice', () => {
  describe('正常情况', () => {
    it('基本价格计算', () => {
      const items = [{ price: 100, quantity: 2 }];
      expect(calcPrice(items, false)).toBe(200);
    });
  });

  describe('折扣', () => {
    it('会员打9折', () => {
      const items = [{ price: 100, quantity: 1 }];
      expect(calcPrice(items, true)).toBe(90);
    });

    it('订单超过1000再减50', () => {
      const items = [{ price: 500, quantity: 2 }];
      expect(calcPrice(items, false)).toBe(950); // 1000 - 50
    });
  });

  describe('边界情况', () => {
    it('空数组返回0', () => {
      expect(calcPrice([], false)).toBe(0);
    });

    it('数量为0返回0', () => {
      const items = [{ price: 100, quantity: 0 }];
      expect(calcPrice(items, false)).toBe(0);
    });
  });
});
```

### Step 4：逐步重构

**从小处开始，一次只改一件事：**

```
这个文件我想分几步重构：

第一步（这次）：
- 把 a、b、c 这样的变量名改成有意义的名称
- 其他不改

请只做这步，改完我跑测试验证。
```

```
第二步（这次）：
- 把这个 200 行的函数拆成几个小函数
- 保持输入输出不变
- 其他不改
```

```
第三步（这次）：
- 提取魔法数字（1000、50、0.9）为常量
- 用常量名替代
```

**不要一次想把所有问题都改掉——容易出 bug，难以定位问题。**

## 🔍 遗留代码常见问题处理

### 问题 1：全局变量满天飞

```javascript
// 重构前
let currentUser = null;
let orderCache = {};

function login(u, p) { currentUser = u; }
function getOrder() { return orderCache[currentUser.id]; }
```

**重构指令：**

```
把全局变量 currentUser 和 orderCache 改成参数传递。
如果需要共享状态，用一个 context 对象包裹。
```

---

### 问题 2：深嵌套回调

```javascript
// 重构前
function getData(id, callback) {
  db.get(id, function(err, data) {
    if (err) callback(err);
    else {
      process(data, function(err, result) {
        if (err) callback(err);
        else save(result, callback);
      });
    }
  });
}
```

**重构指令：**

```
把这个回调风格改成 async/await。
保持相同的输入输出和错误处理逻辑。
```

---

### 问题 3：字符串拼接 SQL

```javascript
// 重构前
const sql = "SELECT * FROM users WHERE id = " + userId + " AND status = '" + status + "'";
```

**重构指令：**

```
改成参数化查询，防止 SQL 注入：
- 用 ? 占位符
- 传入参数数组
- 保持查询逻辑不变
```

**重构后：**

```javascript
const sql = "SELECT * FROM users WHERE id = ? AND status = ?";
const result = await db.query(sql, [userId, status]);
```

---

### 问题 4：重复代码

**重构指令：**

```
这个文件有多处重复的验证逻辑：
- validateEmail()
- validatePhone()
- validatePassword()

把它们提取成一个通用的 validate(value, rules) 函数，
支持配置化规则（必填、格式、最小长度等）。
```

## 🛡️ 遗留代码改造安全手册

| 原则 | 做法 |
|------|------|
| **先测后改** | 用 git stash 暂存，改完 pop 回去对比 |
| **小步提交** | 每个小改动单独 commit，标记 "refactor(legacy): ..." |
| **对比测试** | 改前改后跑相同的测试，确认结果一致 |
| **保留历史** | 不要一次性删除旧代码，用注释标记废弃 |
| **有记录** | 记录踩的坑和做的决策，补充到代码注释 |

## 📝 小结

改造遗留代码的核心：

1. **先理解，再动手** — 花时间看懂代码，比盲目改更值
2. **文档即测试** — 写注释的过程就是理清思路的过程
3. **测试是护身符** — 没有测试的重构是裸奔
4. **小步快跑** — 一次只改一个问题，改完就验证

**改造遗留代码是一场马拉松，不是百米冲刺。保持耐心，稳步前进。**
